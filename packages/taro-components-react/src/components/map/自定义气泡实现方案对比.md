# 自定义气泡实现方案对比

> **H5 端 Map 组件自定义气泡的实现方案评估**
> 创建时间: 2026-03-26
> 适用场景: Taro H5 端，需要对齐小程序 CoverView slot="callout" 行为

---

## 📋 核心需求

1. **业务代码完全无感知** - 小程序和 H5 使用相同的写法
2. **React 组件完整支持** - MapItemMark 等业务组件零改动
3. **事件系统正常工作** - 子元素的 onClick 等事件正常触发
4. **性能可接受** - 支持数十个 marker 的场景

---

## 🎯 方案对比

### 方案 1: absolute View (当前方案 ✅)

```tsx
<View style={{ position: 'absolute', transform: `translate(${x}px, ${y}px)` }}>
  {children}
</View>
```

**实现原理:**
- 在地图容器外创建绝对定位的 View 容器
- 通过 `map.projectToContainer()` 将经纬度转换为屏幕坐标
- 监听地图事件 (`bounds_changed`, `zoom_changed`, `drag`, `rotate`) 实时更新位置

**优势:**
- ✅ React 组件树完整，业务组件零改动
- ✅ 事件系统正常工作，onClick 等事件正常冒泡
- ✅ 代码简单，易于理解和调试
- ✅ 纯 React 实现，无额外依赖

**劣势:**
- ❌ 手动计算坐标转换
- ❌ 手动监听多个地图事件
- ❌ 每次地图变化都触发 DOM 操作

**性能:**
- ⭐⭐⭐ (中等) - 数十个 marker 无明显问题

**适用场景:**
- 复杂的 React 组件树
- 需要完整的事件系统
- 业务代码无感知优先

---

### 方案 2: DOMOverlay

```tsx
class CustomCallout extends TMap.DOMOverlay {
  onPosition(position) {
    this.dom.style.transform = `translate(${position.x}px, ${position.y}px)`
  }

  onInsert() {
    this.dom = document.createElement('div')
    this.dom.innerHTML = content  // ← 问题: 只能是 HTML 字符串
    this.getParent().appendChild(this.dom)
  }
}
```

**实现原理:**
- 继承 `TMap.DOMOverlay` 创建自定义覆盖物
- 地图引擎自动调用 `onPosition()` 更新位置
- 需要通过 `ReactDOM.createPortal` 或 `ReactDOM.render` 渲染 React 组件

**优势:**
- ✅ 地图引擎自动处理坐标转换
- ✅ 地图引擎自动优化位置更新（批量更新、防抖等）
- ✅ 性能优于 absolute View

**劣势:**
- ❌ 需要包装 React 组件，增加复杂度
- ❌ 生命周期对接复杂 (onInsert/onRemove ↔ mount/unmount)
- ❌ 调试困难，多一层抽象
- ❌ 事件系统需要手动对接 React 和 DOM

**性能:**
- ⭐⭐⭐⭐ (良好) - 地图引擎内部优化

**适用场景:**
- 性能敏感场景
- 可以接受额外的封装复杂度
- 简单的 DOM 结构

---

### 方案 3: InfoWindow (原生组件)

```tsx
const infoWindow = new TMap.InfoWindow({
  map,
  position: new TMap.LatLng(lat, lng),
  content: '<div class="custom-callout">...</div>',  // ← 纯 HTML
})
```

**实现原理:**
- 使用腾讯地图原生的 InfoWindow 组件
- 通过 `content` 属性传入 HTML 字符串

**优势:**
- ✅ 地图原生支持，性能最优
- ✅ 自动坐标转换和位置更新
- ✅ 代码最简单

**劣势:**
- ❌ 只支持 HTML 字符串，不支持 React 组件
- ❌ 样式定制受限
- ❌ 无法复用 MapItemMark 等业务组件
- ❌ 事件处理复杂（需要全局监听或 eval）

**性能:**
- ⭐⭐⭐⭐⭐ (最优)

**适用场景:**
- 简单的文本提示框
- 不需要复杂交互的场景

---

### 方案 4: CSS Overlay (fixed 定位)

```tsx
<View style={{ position: 'fixed', left, top }}>
  {children}
</View>
```

**实现原理:**
- 使用 `fixed` 定位，相对于视口定位
- 手动计算屏幕坐标

**优势:**
- ✅ React 组件树完整
- ✅ 事件系统正常

**劣势:**
- ❌ 和 absolute View 类似，需要手动计算坐标
- ❌ `fixed` 在某些场景下可能有滚动问题
- ❌ 不如 absolute 定位灵活

**性能:**
- ⭐⭐⭐ (中等) - 和 absolute View 类似

**适用场景:**
- 特殊布局需求
- 作为 absolute View 的替代方案

---

### 方案 5: Marker 自定义图标 (Canvas/SVG)

```tsx
<MultiMarker
  geometries={[{
    styleId: 'custom-marker',
    position: { lat, lng },
  }]}
  styles={{
    'custom-marker': {
      width: 100,
      height: 50,
      anchor: { x: 50, y: 50 },
      // ← 用 Canvas 或 SVG 绘制气泡
    },
  }}
/>
```

**实现原理:**
- 用 Canvas 或 SVG 绘制气泡图标
- 通过 MultiMarker 批量渲染

**优势:**
- ✅ 性能最优，地图引擎渲染
- ✅ 支持复杂图形
- ✅ 批量更新，适合大量 marker

**劣势:**
- ❌ 不能使用 React 组件
- ❌ 需要用 Canvas/SVG API 绘制
- ❌ 事件处理复杂（需要判断点击坐标）
- ❌ 动态内容更新困难

**性能:**
- ⭐⭐⭐⭐⭐ (最优)

**适用场景:**
- 大量 marker (100+)
- 不需要复杂交互
- 气泡内容固定或简单

---

### 方案 6: SVG Overlay

```tsx
<svg style={{ position: 'absolute', width: '100%', height: '100%' }}>
  <foreignObject x={x} y={y} width={200} height={100}>
    <div xmlns="http://www.w3.org/1999/xhtml">
      {children}
    </div>
  </foreignObject>
</svg>
```

**实现原理:**
- 用 SVG 的 `foreignObject` 嵌入 HTML
- 通过 SVG 的坐标系统定位

**优势:**
- ✅ React 组件树完整
- ✅ 可以嵌入 HTML

**劣势:**
- ❌ SVG 层级问题
- ❌ 性能不如原生方案
- ❌ 复杂度高

**性能:**
- ⭐⭐ (较差)

**适用场景:**
- 特殊图形需求
- 需要 SVG 特性

---

### 方案 7: Canvas 渲染

```tsx
<canvas ref={canvasRef} />
// 用 canvas 2D context 绘制气泡
```

**实现原理:**
- 用 Canvas 绘制所有气泡
- 手动处理点击事件（坐标判断）

**优势:**
- ✅ 性能最优（单一 Canvas）
- ✅ 完全自定义渲染

**劣势:**
- ❌ 完全不能用 React 组件
- ❌ 事件处理需要手动实现
- ❌ 文字排版、样式需要自己实现

**性能:**
- ⭐⭐⭐⭐⭐ (最优)

**适用场景:**
- 需要完全自定义渲染引擎
- 极端性能要求

---

### 方案 8: Web Components

```tsx
customElements.define('custom-callout', class extends HTMLElement {
  connectedCallback() {
    this.attachShadow({ mode: 'open' })
    // 渲染内容
  }
})
```

**实现原理:**
- 用 Shadow DOM 封装自定义元素
- 通过 Web Components 标准

**优势:**
- ✅ 样式隔离
- ✅ 跨组件库复用

**劣势:**
- ❌ 需要包装 React 组件
- ❌ 复杂度高
- ❌ 浏览器兼容性

**性能:**
- ⭐⭐⭐ (中等)

**适用场景:**
- 需要样式隔离
- 跨组件库复用

## 🎯 方案选择建议

### 当前场景 (业务代码无感知 + React 组件 + 复杂交互)

**推荐: absolute View** ✅

**原因:**
1. React 组件树完整，MapItemMark 等业务组件零改动
2. 事件系统正常，onClick 等事件正常冒泡
3. 代码简单，易于理解和调试
4. 性能足够（数十个 marker 无明显问题）

---

### 如果后续有性能问题

**次选: DOMOverlay + ReactDOM.createPortal** 🔶

**实现思路:**
```tsx
class CustomCallout extends TMap.DOMOverlay {
  onInsert() {
    this.dom = document.createElement('div')
    this.dom.style.position = 'absolute'
    this.getParent().appendChild(this.dom)

    // 用 Portal 渲染 React 组件
    ReactDOM.createPortal(
      <MapItemMark {...this.props} />,
      this.dom
    )
  }
}
```

**需要解决:**
1. 生命周期对接 (onInsert/onRemove ↔ mount/unmount)
2. Props 更新同步
3. 事件系统对接

**预期收益:**
- 性能提升 20-30%
- 地图引擎自动优化位置更新

---

### 大量 marker 场景 (100+)

**推荐: Marker 自定义图标** + **单个 InfoWindow 展示详情**

**实现思路:**
1. 用 Canvas/SVG 绘制所有 marker 的简化图标
2. 点击 marker 时，弹出单个 InfoWindow 展示完整信息

**预期收益:**
- 性能提升 80%+
- 支持数百个 marker

---

## 📝 决策记录

**2026-03-26: 选择 absolute View 作为当前方案**

- 理由: 业务代码无感知优先，React 组件完整支持，复杂度低
- 预留优化空间: 后续可迁移到 DOMOverlay
